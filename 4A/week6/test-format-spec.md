# Test Case Format Specification

## Overview
This document defines the standard format for all Week 6 test cases for the JKYog WhatsApp Bot system.  
All test cases are designed to align with Team 4B’s API contract and Twilio webhook behavior.

---

## Content Type

### Webhook Requests
All webhook requests must use:

Content-Type: application/x-www-form-urlencoded

---

## Webhook Format

### Endpoint
POST /webhook/whatsapp

### Body Format (form-encoded)
From=whatsapp:+1234567890&Body=Hello&ProfileName=Test User

---

## Required Fields

| Field        | Description                  |
|-------------|------------------------------|
| MessageSid  | Unique message identifier from Twilio |
| From        | Sender WhatsApp number       |
| Body        | User message content         |
| ProfileName | WhatsApp display name        |

---

## Authentication Rules

### 1. Webhook Authentication (Twilio)
- Uses X-Twilio-Signature header
- Verified using server-side TWILIO_AUTH_TOKEN
- This is the ONLY authentication method for webhook requests
- Ensures the request originated from Twilio

Header:
X-Twilio-Signature: <valid-signature>

---

### 2. Other Endpoints Authentication
- No Bearer token authentication is used in this system
- Authentication (if required) is handled internally by backend services through Twilio
- Most endpoints in this project do not require authentication

---

### 3. Create Payment Endpoint
Endpoint: POST /create-payment

- Supports guest checkout
- NO authentication required
- NO bearer token
- NO signature required

---

## Expected Responses

| Status Code | Meaning                          |
|------------|----------------------------------|
| 200        | Successful request               |
| 400        | Bad request / invalid input      |
| 401        | Authentication failure           |
| 404        | Endpoint not found               |
| 500        | Internal server error            |

---

## Validation Rules

- All webhook requests MUST:
  - Be form-encoded
  - Include MessageSid,From, Body, and ProfileName
  - Include a valid X-Twilio-Signature header (for authenticated scenarios)

- Requests missing required fields should return:
  - 400 Bad Request

- Requests with missing or invalid signature should return:
  - 401 Unauthorized

---

## Notes

- MessageSid is automatically generated by Twilio and should NOT be manually included in test cases
- Encoding format example:
  - whatsapp:+1234567890 → whatsapp%3A%2B1234567890 (URL encoded)
- All test cases must follow the structure defined in this document
- This specification ensures consistency across all test scenarios and enables reliable cross-team integration testing
